# Lessons（用户更正后沉淀的规则）

## 2026-07-18 · 悬停提示与判定规则必须同一提交内同步（用户87号更正）
**规则**：任何改动判定逻辑/阈值的提交（signal.config.js、signal.js、server 判定链），必须在**同一个提交**里同步：
1. frontend/src/i18n 七语言的 hints.* / hintGlobal / interpret.* / 档位描述；
2. skills/stock-sentinel/SKILL.md；
3. backend/api/mcp.js 与 mcp/index.js 的工具描述。
**自查方法**：改完规则后 grep 前端 locales 里涉及该维度的所有键逐条比对。历史教训：v2 判定上线后
七语言文案滞后了一天才被一致性审查发现（渐进加息旧文案），W5 上线时 defense 描述又漏了趋势例外。

**2026-07-20 扩展（103号审查教训）**：同步范围补两处更隐蔽的：
4. README.md 的信号计算表与全局叠加段——README 停在 MVP 三档/SMH-SPY 时代历经六轮判定迭代
   无人发现，外部访客看到的第一页与系统实际行为差了一代；
5. backend/api/daily-report.js 的 buildFacts 判定规则行——这是每天喂给 LLM 的解读依据，
   旧规则"进攻=四维全宽松"导致日报"距进攻还差什么"的解读一直基于错误规则。
**为什么隐蔽**：界面文案有用户天天看着，README/日报规则行没有日常曝光路径，只能靠提交纪律。

## 2026-07-18 · 重大新判定输入不能只靠"一并执行"式的批量授权（用户88号质疑）
**规则**：凡是**新增参与判定的输入**（哪怕只是否决器，如收益率曲线倒挂否决进攻档），上线前要在
回复里单独一段醒目说明"新增了什么判定角色、为什么、如何撤销"，并等用户看到；批量授权
（"其他的也一并执行"）只适用于修 bug、文案、口径对齐类，不适用于引入新判定维度。
历史教训：收益率曲线否决器随"一并执行"上线，用户事后表示"没有印象"。

## 2026-07-18 · 用户目标显著超出数学可行域时，先算天花板再讨论方案（用户89号）
**规则**：遇到"年化100%"类目标，不直接说不可能也不假装可行——先用回测数据算出该框架的
理论上限（如完美预知择时的年化），把目标、现状、天花板放在同一张表里给用户看，再给出
逼近天花板的现实路径（换标的/杠杆/集中度）及其代价。用户自己的 CLAUDE.md 要求"太离谱要指出来"。
